Lab88--Zaawansowane-JPA-and-Audyt-Historyczny-Hibernate-Envers--Pełna-historia-zmian-rekordów
=============================================================================================

Temat, który jest absolutnym wymogiem w systemach finansowych i ubezpieczeniowych (tzw. Compliance).

Problem: 
Obecnie mamy JPA Auditing - Lab 81, który mówi nam: 
"Jacek zmienił tę transakcję 5 minut temu". 
Ale nie wiemy, co Jacek zmienił. 
Czy zmienił kwotę z 100 na 1000? 
Czy zmienił kategorię? 
Stary stan rekordu przepada na zawsze.

Rozwiązanie: 
`Hibernate Envers`. 
To narzędzie automatycznie tworzy tabele historyczne (tzw. _AUD), 
w których zapisuje każdą wersję rekordu. 
To pozwala na "podróż w czasie" i sprawdzenie, jak transakcja wyglądała rok temu.

Krok-1. Dodanie zależności (build.gradle)
-----------------------------------------
Envers to moduł Hibernate'a. Dodaj go do dependencies:

```groovy
// Hibernate Envers - wersjonowanie danych
implementation 'org.hibernate.orm:hibernate-envers'
```
(Odśwież Gradle!)

Krok-2. Włączenie wersjonowania w Encjach
-----------------------------------------
Musimy wskazać, które tabele mają być śledzone. Zazwyczaj chcemy śledzić transakcje.

Zaktualizuj `src/main/groovy/pl/edu/praktyki/repository/TransactionEntity.groovy`:

```groovy
package pl.edu.praktyki.repository

import jakarta.persistence.*
import org.hibernate.envers.Audited // <--- DODAJ IMPORT
import org.springframework.data.annotation.*
import org.springframework.data.jpa.domain.support.AuditingEntityListener
import java.time.LocalDateTime

@Entity
@Table(name = "transactions")
@EntityListeners(AuditingEntityListener.class)
@Audited // <--- MAGIA: Od teraz każda zmiana w tej tabeli trafi do tabeli transactions_aud
class TransactionEntity {
// ... reszta pól bez zmian ...
}
```

Jeśli oznaczyłeś klasę TransactionEntity jako `@Audited`, to każda encja, 
z którą jest ona powiązana (Twoja `CategoryEntity`), również musi być audytowalna. 
`Envers` nie może zapisać historii transakcji, nie wiedząc, jak zapisać historię powiązanej z nią kategorii.

Audytuj również kategorie
To najlepsze podejście. 
Chcemy wiedzieć nie tylko, że transakcja się zmieniła, 
ale też czy np. nie zmieniono jej kategorii na taką z innym limitem.

Dodaj @Audited do CategoryEntity.groovy

```groovy
   @Entity
   @Table(name = "categories")
   @Audited // <--- DODAJ TO
   @Canonical
   class CategoryEntity { ... }
```
 

Krok-3. Migracja Flyway (V10__add_envers_tables.sql  oraz  V11__add_category_audit_table.sql)
---------------------------------------------------------------------------------------------

A. Dla tabeli Transactions:
---------------------------
Envers potrzebuje dwóch tabel systemowych: jednej na metadane rewizji i jednej na kopie Twoich danych.

`src/main/resources/db/migration/V11__add_envers_tables.sql`:

```sql
-- 1. Tabela informacji o rewizji (systemowa dla Enversa)
CREATE TABLE revinfo (
                         rev INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
                         revtstmp BIGINT
);

-- 2. Tabela historyczna (Audit Table)
-- Musi zawierać kolumny z oryginalnej tabeli + pola Enversa (rev, revtype)
CREATE TABLE transactions_aud (
                                  db_id BIGINT NOT NULL,
                                  rev INTEGER NOT NULL,
                                  revtype SMALLINT, -- ZAMIAST TINYINT (0: ADD, 1: MOD, 2: DEL)

                                  original_id VARCHAR(255),
                                  date DATE,
                                  amount NUMERIC(19, 2),
                                  currency VARCHAR(10),
                                  amountpln NUMERIC(19, 2),
                                  category VARCHAR(255),
                                  description VARCHAR(255),
                                  category_id BIGINT,

    -- Uwaga: Pól technicznych audytu (created_date itp.) zazwyczaj się nie wersjonuje,
    -- bo one same w sobie są informacją o czasie. Zostawiamy jak wyżej.

                                  PRIMARY KEY (db_id, rev),
                                  CONSTRAINT fk_transactions_aud_revinfo FOREIGN KEY (rev) REFERENCES revinfo(rev)
);
```

B. Dla tabeli Categories:
-------------------------
Musisz stworzyć tabelę historyczną dla kategorii.

Dodaj nową migrację Flyway: `src/main/resources/db/migration/V11__add_category_audit_table.sql`:

```sql
CREATE TABLE categories_aud (
id BIGINT NOT NULL,
rev INTEGER NOT NULL,
revtype SMALLINT,
name VARCHAR(50),
monthly_limit DECIMAL(19, 2),
PRIMARY KEY (id, rev),
CONSTRAINT fk_categories_aud_revinfo FOREIGN KEY (rev) REFERENCES revinfo(rev)
);
```


Krok-4. Test Spock – "Wehikuł Czasu" (EnversSpec.groovy)
--------------------------------------------------------
To wyzwanie dla Ciebie. 
Sprawdzimy, czy po edycji transakcji, w bazie zostaną dwa wpisy: stary i nowy.

`src/test/groovy/pl/edu/praktyki/repository/EnversSpec.groovy`:

```groovy
package pl.edu.praktyki.repository

import org.springframework.beans.factory.annotation.Autowired
import pl.edu.praktyki.BaseIntegrationSpec
import org.springframework.transaction.annotation.Transactional

class EnversSpec extends BaseIntegrationSpec {

    @Autowired TransactionRepository repository
    @Autowired jakarta.persistence.EntityManager entityManager

    def "powinien zachować historię zmian kwoty transakcji"() {
        given: "transakcja z kwotą 100"
        def tx = new TransactionEntity(originalId: "HIST-1", amountPLN: 100.0)
        tx = repository.saveAndFlush(tx)
        def id = tx.dbId

        when: "zmieniamy kwotę na 200"
        tx.amountPLN = 200.0
        repository.saveAndFlush(tx)

        then: "w głównej tabeli jest 200"
        repository.findById(id).get().amountPLN == 200.0

        and: "w tabeli Envers (AuditReader) mamy dostęp do obu wersji"
        def reader = org.hibernate.envers.AuditReaderFactory.get(entityManager)
        // Pobieramy listę numerów rewizji dla tego ID
        def revisions = reader.getRevisions(TransactionEntity, id)
        
        revisions.size() == 2 // Jedna dla INSERT, druga dla UPDATE
        
        and: "możemy wyciągnąć pierwszą (historyczną) wersję"
        def oldVersion = reader.find(TransactionEntity, id, revisions[0])
        oldVersion.amountPLN == 100.0
    }
}
```

Dlaczego to jest "Mid-level Hardcore"?

Legal / Compliance: 
W bankowości nikt nie wierzy na słowo. 
Jeśli z konta zniknęły pieniądze, musisz umieć wyciągnąć raport: "Dnia 12 marca o 10:00 kwota wynosiła X, a o 10:05 użytkownik Y zmienił ją na Z".

AuditReader: 
Poznałeś niskopoziomowe API Hibernate'a do czytania danych, które nie są widoczne przez standardowe repozytoria Spring Data.

Data Integrity: 
Zrozumiałeś, jak `Envers` używa tabeli `REVINFO` do śledzenia globalnego czasu zmian w całym systemie.
