Lab78--API-And-Security-Advanced--Database-Authentication-And-Password_Hashing
==============================================================================

To moment, w którym Twój system przestaje wierzyć każdemu na słowo i zaczyna sprawdzać "tożsamość" w swojej bazie danych.
Obecnie Twój system jest "pancerny" technicznie, ale ma jedną słabość: Użytkownicy są "udawani". 
Generujemy token dla "admina" w teście, ale nie mamy prawdziwej tabeli użytkowników w bazie.
Na poziomie Mida musisz wiedzieć, jak przechowywać hasła `BCrypt` i jak ładować użytkowników z bazy danych.

Cel:
1. Stworzenie tabeli `users` w PostgreSQL.
2. Implementacja bezpiecznego hashowania haseł `BCrypt`.
3. Stworzenie endpointu `/api/auth/register`, który pozwoli dodać nowego użytkownika.

Krok-2. Model Użytkownika (UserEntity.groovy)
---------------------------------------------

`src/main/groovy/pl/edu/praktyki/security/UserEntity.groovy`

```groovy
package pl.edu.praktyki.security

import jakarta.persistence.*
import groovy.transform.Canonical

@Entity
@Table(name = "users")
@Canonical
class UserEntity {
   @Id
   @GeneratedValue(strategy = GenerationType.IDENTITY)
   Long id

   @Column(unique = true, nullable = false)
   String username

   @Column(nullable = false)
   String password

   @Column(nullable = false)
   String role
}
```

Krok-4. Spring Security Service (CustomUserDetailsService.groovy)
-----------------------------------------------------------------
To jest "most" łączący Spring Security z Twoją bazą danych.
Spring Security potrzebuje "tłumacza", który na podstawie nazwy użytkownika pójdzie do Twojej bazy
i wyciągnie hasło oraz role.

`src/main/groovy/pl/edu/praktyki/security/CustomUserDetailsService.groovy`

```groovy
package pl.edu.praktyki.security

import org.springframework.beans.factory.annotation.Autowired
import org.springframework.security.core.userdetails.*
import org.springframework.security.core.authority.SimpleGrantedAuthority
import org.springframework.stereotype.Service

@Service
class CustomUserDetailsService implements UserDetailsService {

   @Autowired UserRepository userRepo

   @Override
   UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
      // Szukamy w naszej bazie Postgres
      UserEntity user = userRepo.findByUsername(username)
              .orElseThrow { new UsernameNotFoundException("Nie znaleziono użytkownika: $username") }

      // Mapujemy naszą encję na obiekt UserDetails, który rozumie Spring
      return User.builder()
              .username(user.username)
              .password(user.password) // Tu jest hash z bazy
              .authorities(new SimpleGrantedAuthority(user.role))
              .build()
   }
}
```

Krok-5. Konfiguracja Kodowania Haseł (SecurityConfig.groovy)
------------------------------------------------------------
Musimy powiedzieć Springowi, żeby używał algorytmu `BCrypt`.
Czyli, powiedz Springowi, jak ma porównywać hasła.
Gdy użytkownik wpisze "admin123", Spring użyje `passwordEncoder`, żeby sprawdzić, czy pasuje do hasha w bazie.

Dodaj do `pl/edu/praktyki/security/SecurityConfig.groovy/SecurityConfig.groovy`

```groovy
@Bean
org.springframework.security.crypto.password.PasswordEncoder passwordEncoder() {
return new org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder()
}
```



Dlaczego to jest "Must-have" na Mida?
-------------------------------------

Rekruter zapyta: 
"Jak przechowujesz hasła w bazie?".
Jeśli powiesz: "Szyfruję je", to błąd (szyfr można odwrócić).
Musisz powiedzieć: "Używam solonego hashowania BCrypt". 
To jest jedyna poprawna odpowiedź.

BCrypt:
To standard branżowy.
Nawet jeśli haker ukradnie Twoją bazę danych, nie pozna haseł użytkowników,
bo odwrócenie hasha BCrypt jest niemal niemożliwe przy dzisiejszej mocy obliczeniowej.

UserDetailsService:
To najbardziej profesjonalny sposób integracji Spring Security z bazą.
Nie używasz "sztywnych" użytkowników w pliku .properties.

Flyway + Security:
Twoi użytkownicy są wersjonowani.
Jeśli dodasz nową rolę, robisz to przez plik .sql.
